# QA 끝까지 눌러보기 원칙

> **"존재 확인"은 QA가 아니다. "끝까지 눌러보기"만 QA다.**
> 카드/버튼/링크가 화면에 보인다는 것과 실제로 동작한다는 것은 완전히 다른 이야기다.

---

## Iron Rule

```
QA 시 요소 존재 확인만으로 PASS 판정 금지.
모든 인터랙티브 요소(카드, 버튼, 링크)는 반드시 실제 클릭 후
다음 화면까지 렌더 확인해야 PASS다.
```

---

## 왜 이 룰이 필요한가 — 2026-04-08 사건

**상황**: Product Builder PR #13 재QA 시, 온보딩 완료 후 "내 프로젝트" 카드가 자동 생성된 것을 snapshot으로만 확인하고 QA-Done 판정 → PR 머지.

**실제 버그**: 사용자가 직접 사용해보니 **프로젝트 카드 클릭이 동작 안 함**. 코드 확인 결과:

```tsx
const handleOpen = (id: string) => {
  updateLastOpened.mutate(id);
  // TODO: Navigate to project workspace when available
};
```

`navigate()` 호출이 없었음. 앱의 가장 기본 플로우(프로젝트 열기)가 **데드엔드**였는데 QA가 이를 잡아내지 못함.

**근본 원인**:
- QA 시나리오가 "카드 생성 확인"까지만 있고 "카드 클릭 → 다음 화면 확인"이 없었음
- 사용자 시나리오의 **종료점(끝 클릭)** 이 아닌 **시작점(진입)** 에서 PASS 판정
- `// TODO` 주석이 프로덕션 경로에 있었음에도 코드 리뷰와 브라우저 테스트 모두 놓침

**결과**: PR #13 머지 후 사용자 발견 → 긴급 follow-up PR #14 필요

## PASS 판정 3단계 검증 (필수)

인터랙티브 요소에 대한 QA는 반드시 아래 3가지를 **모두** 확인해야 PASS다:

| # | 검증 | 도구 | 실패 시 |
|---|------|------|:------:|
| 1 | **URL 전환** | `chrome-devtools-mcp navigate_page` 결과 URL 확인 | FAIL |
| 2 | **다음 화면 DOM 렌더** | `chrome-devtools-mcp take_snapshot` (a11y tree) | FAIL |
| 3 | **관련 네트워크 요청 200** | `chrome-devtools-mcp list_network_requests` | FAIL |

**셋 중 하나라도 빠지면 PASS 아님.** 스크린샷 실패 시 accessibility snapshot으로 대체 가능하지만, DOM 구조 검증 자체를 생략할 수는 없다.

> **직접 QA 브라우저 검증은 chrome-devtools-mcp 를 1순위로 사용한다.** `Transport closed`, 도구 미노출, 세션 비활성, 설치 누락 등으로 실패하면 Playwright fallback 을 사용하고, fallback 사유와 검증 시나리오를 보고한다. 상세는 `docs/rules/runtime-verification.md` 의 "브라우저 직접 검증 도구" 섹션 참조.

## 디스패치 전 필수 질문

서브에이전트에게 QA를 시키기 전, Lead(Iris)는 스스로에게 물어야 한다:

1. **"이 플로우의 마지막 클릭은 어디인가?"**
2. **"거기까지 눌러보라고 명시적으로 지시했는가?"**
3. **"'존재 확인'이 아닌 '동작 확인'으로 시나리오를 작성했는가?"**

하나라도 "아니오"면 디스패치 전 시나리오 재작성.

---

## 시나리오 작성 원칙

### ❌ 잘못된 시나리오 (시작점 기준)

```
S1. 로그인 → 대시보드 진입 확인
S2. 온보딩 → 프로젝트 자동 생성 확인
```

"진입 확인"으로 끝나는 것은 시작점 QA. 사용자가 실제로 하는 행동이 아님.

### ✅ 올바른 시나리오 (종료점 기준)

```
S1. 로그인 → 대시보드 진입 → "내 프로젝트" 카드 클릭
    → /p/{id}/lore 진입 확인 → 사이드바 + 필터 탭 렌더 확인 → story.*.list 200 확인
S2. 신규 가입 → 온보딩 3단계 완료 → 자동 생성 프로젝트 클릭
    → 워크스페이스 첫 작업 화면 렌더 확인
```

**마지막 액션이 "클릭 → 다음 화면 검증"**이어야 종료점 QA다.

---

## TODO 주석 = FAIL 플래그

프로덕션 경로의 코드에 아래 패턴이 있으면 **즉시 FAIL**:

| 패턴 | 의미 |
|------|------|
| `// TODO: ...when available` | 미구현 |
| `// TODO: navigate ...` | 네비게이션 누락 |
| `// TODO: implement` | 핸들러 미구현 |
| `// FIXME:` | 알려진 버그 |
| `// HACK:` | 임시 우회 |
| 빈 `onClick={() => {}}` 핸들러 | 동작 없음 |
| `console.log("TODO")` | 미구현 로그 |

코드 리뷰 단계에서 이를 탐지하고, 브라우저 테스트에서 실제 재현하여 2중 확인.

---

## 재현성 검증

동일 플로우를 **2회 이상** 반복하여 1회 성공이 우연이 아님을 확인:

```
1회차: 카드 클릭 → 워크스페이스 진입 → OK
대시보드 복귀 → 재로드
2회차: 카드 클릭 → 워크스페이스 진입 → OK
```

2회차에서 실패하거나 다른 동작을 보이면 **경쟁 상태(race condition)** 의심. 해당 사항 즉시 보고.

## 서브에이전트 프롬프트 템플릿

Zion(Browser Tester) 디스패치 시 아래 문구를 **반드시** 포함:

```
## 끝까지 클릭 원칙 (Iron Rule)

각 시나리오의 마지막 단계는 "존재 확인"이 아닌 "실제 클릭 + 다음 화면 검증"
이어야 합니다. 카드, 버튼, 링크가 화면에 보이는 것만으로는 PASS가 아닙니다.

PASS 3단계 검증:
1. URL 전환 확인
2. 다음 화면 DOM 렌더 확인 (accessibility snapshot)
3. 관련 네트워크 요청 200 확인

하나라도 빠지면 FAIL.

시나리오는 시작점("로그인 성공")이 아닌 종료점("로그인 → 대시보드 → 카드 클릭
→ 워크스페이스 첫 작업 UI 렌더")으로 작성하세요.
```

## 금지 사항

| 금지 | 이유 |
|------|------|
| "카드가 생성된 것 확인" 으로 PASS | 동작 미검증 |
| "로그인 성공" 만으로 인증 시나리오 PASS | 사용자는 로그인 후 뭔가를 함 |
| 스크린샷 없이 "정상 동작" 서술 | 증거 없음 |
| TODO 주석 무시 | 데드엔드 위험 |
| 2회차 재현 생략 | race condition 놓침 |
| 시작점 기준 시나리오 작성 | 종료점 검증 누락 |
