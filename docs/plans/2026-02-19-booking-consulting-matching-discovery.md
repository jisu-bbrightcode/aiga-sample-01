# 예약/상담/매칭 + 결제 Discovery

작성일: 2026-02-19  
단계: Discovery (IDEA 탐색)  
상태: 리뷰 대기

## 1) 한 줄 정의

사용자가 전문가를 탐색/매칭받고, 캘린더에서 상담 시간을 예약한 뒤 선결제로 확정하며, 취소/환불까지 셀프로 처리할 수 있는 서비스형 비즈니스 코어 기능.

## 2) 문제 정의

- 예약 과정이 DM/전화/수기 캘린더로 분산되어 이탈이 발생한다.
- 결제 전 예약 홀드는 노쇼/미입금 리스크를 만든다.
- 취소/환불 기준이 불명확해 CS 비용이 증가한다.
- 상담사-고객 매칭 기준이 개인 경험에 의존해 품질 편차가 크다.

## 3) 사용자 및 핵심 JTBD

- 고객(최종 사용자): "내 상황에 맞는 상담사를 빠르게 찾고, 바로 결제해 시간을 확정하고 싶다."
- 상담사/서비스 제공자: "가용 시간을 공개하고 예약/변경/취소를 자동으로 처리하고 싶다."
- 운영자: "환불/분쟁을 정책 기반으로 처리하고 운영 리스크를 낮추고 싶다."

## 4) 가치 가설

- 사용자 가치
- 탐색부터 결제까지 단일 플로우로 예약 확정 시간을 단축한다.
- 사전 결제로 일정 확정성과 신뢰를 높인다.
- 셀프 취소/환불로 사용자 통제감을 높인다.
- 비즈니스 가치
- 예약 확정률 상승, 노쇼율 감소.
- 선결제 기반 매출 인식 시점 앞당김.
- 운영팀 수동 처리(정산/환불/일정 조정) 비용 절감.

## 5) MVP 범위 (Discovery 제안)

### In Scope (P0)

- 상담사 프로필/전문분야/요금/가능 시간 슬롯 노출
- 예약 캘린더(일/주 단위 슬롯 선택, 중복 예약 방지)
- 선결제 체크아웃(예약 생성과 원자적 결합)
- 정책 기반 취소/환불
- 예: 시작 24시간 이전 100%, 24시간 이내 50%, 노쇼 0%
- 예약 상태 머신
- `pending_payment` -> `confirmed` -> `completed`
- `cancelled_by_user` / `cancelled_by_provider` / `refunded`
- 알림
- 예약 확정, 리마인더, 취소/환불 결과

### In Scope (P1)

- 규칙 기반 매칭 v1
- 조건: 카테고리, 예산, 가능한 시간, 언어, 상담 방식(온라인/오프라인)
- 상위 N개 추천 + 추천 근거 표시

### Out of Scope (이번 사이클 제외)

- 실시간 영상통화 내장(WebRTC)
- 동적 가격(시간대/수요 기반)
- 다자간 그룹 세션, 기업 계약/법인 청구
- 고도화 ML 매칭 모델(학습 기반 랭킹)

## 6) 기존 자산 재사용 관점 (레포 기준)

- 결제: `payment` feature의 체크아웃/환불/구독 인프라 재사용 가능
- 알림: `notification` feature + 실시간/설정 체계 재사용 가능
- 사용자/권한: `profile`, `role-permission` 재사용 가능
- 스케줄링: `scheduled-job` 및 기존 캘린더 패턴 재사용 가능
- 캘린더 UI/패턴: `content-studio`, `marketing`의 캘린더 UX 참고 가능

## 7) 핵심 리스크 및 검증 포인트

- 결제-예약 원자성 실패 시 데이터 불일치 위험
- 대응: 예약 생성 트랜잭션 + 결제 콜백 기반 상태 전이 설계 필요
- 환불 정책 예외(상담사 귀책/시스템 장애) 처리 기준 필요
- 매칭 설명 가능성 부족 시 사용자 신뢰 저하
- 시간대(Timezone) 처리 오류 시 운영 사고 가능성 큼

## 8) Discovery 가정

- 비즈니스 모델은 "건당 선결제"가 1차 수익모델이다.
- 상담 단위는 기본 1회/단일 슬롯이다.
- 초기 타깃은 B2C 또는 1:1 상담형 서비스다.
- 결제 수단/PG는 기존 `payment` feature가 다루는 체계를 따른다.

## 9) 성공 지표 (초안)

- 예약 요청 -> 결제 완료 전환율
- 결제 완료 -> 실제 상담 완료율
- 취소율/노쇼율
- 환불 처리 리드타임
- 매칭 추천 클릭률/예약 기여율

## 10) PRD 진입 전 결정 필요 사항

- 매칭 방식 확정
- 자동 추천만 vs 추천 + 사용자가 직접 필터 검색
- 결제 정책 확정
- 즉시 결제만 vs 홀드 후 제한시간 결제
- 취소/환불 정책 확정
- 표준 정책 1개 vs 상담사별 커스텀 정책
- 공급자 온보딩 범위
- 상담사 셀프 시간표 관리 포함 여부
- 상담 수행 형태
- 온라인 링크 외부 연동(Zoom/Meet) 포함 여부

## 11) Discovery 결론

- 본 기능은 "부가 기능"이 아니라 서비스형 비즈니스의 매출 코어 플로우에 해당한다.
- 현재 코드베이스 자산(payment/notification/profile/scheduled-job)을 재사용하면 신규 구현 난이도는 높지만 MVP를 빠르게 구성할 수 있다.
- PRD 단계로 진입할 가치가 충분하며, 상기 결정사항 확정 후 PRD 작성 권장.
